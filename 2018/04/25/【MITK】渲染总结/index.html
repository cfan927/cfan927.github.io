【MITK】渲染总结

发表于 2018-04-25 | 分类于 mitk

编辑：王祺 审校：田兆瑞

1.综述

官方文档参考：http://docs.mitk.org/nightly/QVTKRendering.html MITK的渲染底层是基于VTK的，大家对VTK的渲染应该都有了解，是一个面向对象的管道过滤器模型的渲染方式。本文建立在大家对VTK的渲染基础有了解的情况下进行讨论的。在VTK中，所有能够存在于一个渲染场景(2D或3D)中的可见物体，都继承自vtkProp(或其子类vtkActor)。MITK使用新的接口类，vtkMitkRenderProp，同样继承自vtkProp，所有的MITK渲染工作都通过这个接口类进行。

2. MITK渲染管线（pipline）

2.1 综述

vtkMitkRenderProp类连接上mitk::VtkPropRender类可以把MITK的渲染管线集成到VTK的渲染管线里。QmitkRenderWindow并不是继承自mitk::RenderWindow，而是继承自QVTKWidget，这个是由VTK提供的，其实就是VTK中的RenderWindow在QT窗口组件中的显示，不考虑QT的话QVTKWidget和RenderWindow没有本质差别。 MITK的渲染流程如下： • 1.向QmitkRenderWindow发送的一个render请求 • 2.更新请求应该被送给RenderingManager • 3. RenderingManager会请求更新所有的vtkRenderWindow • 4. vtkRenderWindow然后调用它关联着的所有vtkRenderer的Render()函数。 • 5. vtkRenderer调用vtkMitkRenderProp中的四个函数 • 6.这些函数调用被转给mitk::VtkPropRenderer • 7.根据mapper的类型(基于OpenGL或VTK)，OpenGL被enabled或disabled。 • 8-1.在OpenGL渲染的情况下，每个mapper的Paint()函数都被调用 • 8-2.如果mapper是基于VTK的，这四个函数调用被转给 mitk::VtkMapper，然后在具体的mapper中实现这四个函数，对应的VtkProp被计算 MITK使用一些特别的vtkRenderer（不在标准MITK渲染管道之中）来渲染一些特殊的物体，比如显示梯度背景(mitk::GradientBackground)、显示视频资源(QmitkVideoBackround和mitk::VideoSource)、或显示logo（mitk::ManufacturerLogo）等等。

2.2.1 QmitkStdMultiWidget

这个类是workbench显示的四个窗口的类，从类图中我们可以看到这个类对应了4个QmitkRenderWindow类，所以它其实是对QmitkRenderWindow（渲染窗口）的集成，主要的渲染的实现以及设置是由下面的类控制的。获取4个QmitkRenderWindow可以通过QmitkRenderWindow * GetRenderWindow (unsigned int number)得到每个QmitkRenderWindow。其中的参数为渲染窗口的编号。 这个类同时开放了一些接口更改集成的4个窗口的具体显示，可以用以下的函数来设置相应的细节和功能：

  • void RequestUpdate ()
    请求窗口更新，一般是对渲染做了改变之后使用

  • void ForceImmediateUpdate ()
    请求立即更新，会强迫render立即渲染，上一个函数是发送一个更新的事件，由管线决定什么时候更新。一般不会使用这个函数

  • mitk::MouseModeSwitcher * GetMouseModeSwitcher ()
    返回一个鼠标模式切换器，这个切换器可以在不同的鼠标交互方式中切换，具体见MouseModeSwitcher类

  • const mitk::Point3D GetCrossPosition () const 得到三种视图平面的交汇点

  • int GetLayout () const
    得到4个窗口的布局，具体的布局是一个枚举，见下图

  • void SetDataStorage (mitk::DataStorage *ds)
    设置4个渲染窗口的datastorage

  • void ActivateMenuWidget (bool state)
    是否打开菜单

  • void SetCornerAnnotationVisibility (bool visibility)
    是否可见中央的注释

  • void SetDecorationProperties (std::string text, mitk::Color color, int widgetNumber)
    设置每个窗口的中央注释，第一个参数为注释内容，第二个为注释的颜色，第三个是要设置的窗口编号

  • void SetGradientBackgroundColorForRenderWindow (const mitk::Color &upper, const mitk::Color &lower, unsigned int widgetNumber)
    设置每个渲染窗口的渐变背景颜色，第一个参数为上方的颜色，第二个位下方的颜色，第三个为窗口编号

  • void SetDecorationColor (unsigned int widgetNumber, mitk::Color color) 设置每个窗口边框的颜色以及注释的颜色

2.2.2 QmitkRenderWindow

这个类在上文已经介绍过继承关系，其实可以理解为继承自VTK的Renderwindow，同时继承自RenderWindowBase。 构造函数：

QmitkRenderWindow (QWidget *parent=nullptr, const QString &name="unnamed renderwindow", mitk::VtkPropRenderer *renderer=nullptr, mitk::RenderingManager *renderingManager=nullptr, mitk::BaseRenderer::RenderingMode::Type renderingMode=mitk::BaseRenderer::RenderingMode::Standard)

从构造函数的参数可以看出来它需要一个QT的窗口来放置这个RenderWindow，需要一个名字，需要一个VtkPropRender，但是可以暂时设置为NULL，还需要一个RenderManager，暂时也可以设置为NULL，最后一个参数是设置这个渲染窗口的渲染模式，默认为Standard，还可以选择为DepthPeeling（深度剥离），MultiSampling（多重采样），在构造函数的函数体中对三种渲染模式进行了相关参数的设置。 函数体中调用了RenderWindowBase的Initialization（）函数来初始化构造函数中的DataStorage、VTKPropRender和vtkMitkRenderProp。这也是为什么在构造函数参数列表中这两个参数可以设置为NULL，因为要在Initialization中进行初始化。通过这个构造函数可以发现渲染窗口已经包含了VtkPropRender和vtkMitkRenderProp。还有一些函数可以对渲染窗口进行设置，下面就来介绍一下。

  • void QmitkRenderWindow::SetResendQtEvents ( bool resend ) 设置渲染窗口接收到的QT事件（例如鼠标点击、键盘按键等等）是否传递给它的父窗口，默认为是。

  • void SetLayoutIndex (unsigned int layoutIndex) 设置窗口的布局索引

  • void QmitkRenderWindow::LayoutDesignListChanged(int layoutDesignIndex) 改变窗口菜单中的布局的列表

  • void HideRenderWindowMenu () 隐藏菜单

  • vtkRenderWindow * GetVtkRenderWindow () override 得到关联的Vtk渲染窗口

  • vtkRenderWindowInteractor * GetVtkRenderWindowInteractor () override 得到关联的Vtk渲染窗口交互

  • void FullScreenMode (bool state)
    是否全屏显示

  • 继承自它的父类mitk::RenderWindowBase的函数，virtual mitk::VtkPropRenderer * GetRenderer ()
    可以得到VtkPropRender。

2.2.3 vtkMitkRenderProp

这个类是个客制化的vtkProp类，它实现了MITK和VTK之间的渲染接口。它把render（）请求传递到VtkPropRenderer，VtkPropRender回应渲染数据的请求。总的来说就是vtkMitkRenderProp定义要渲染的物体，然后VtkPropRender定义如何渲染，这两个对应的VTK管线中的Prop和Render。 下面是这个类开放的公共接口。

  • void SetPropRenderer (mitk::VtkPropRenderer::Pointer propRenderer)
    为这个vtkMitkRenderProp设置对应的VtkPropRender

  • int RenderOpaqueGeometry (vtkViewport *viewport) override 渲染不透明的几何部分

  • int RenderOverlay (vtkViewport *viewport) override
    渲染遮盖的几何部分

  • double * GetBounds () override
    得到渲染窗口的边界（大小）。

  • void ReleaseGraphicsResources (vtkWindow *window) override 调 用了vtkPropRender里同名的函数，会在那个类里介绍。

  • void InitPathTraversal () override vtkAssemblyPath * GetNextPath () override int GetNumberOfPaths () override 这三个函数都是VtkPicker相关的函数，API Reference中写的是前两个函数会提供给vtkPicker一个包含所有MITK对象的列表和所有基于VTK的mapper。首先介绍vtkPicker：vtkPicker用于拾取vtkProp3D实例，它通过发射一条光线到图形窗口中，并与Actor的线框(Bounding box)相交，光线是从窗口坐标系统中拾取的一点到相机所在的位置的连线。因为光线可能会与多个Actor的线框相交，所以可能会返回多个vtkProp3D的实例。vtkPicker返回的是与光线相交的Prop实例的一个列表、世界坐标系统中的拾取坐标以及最靠近相机的Prop和Mapper实例。所以个人认为这几个函数是用来告诉vtkPicker可能拾取到的物体的所有类型和所有可能的Mapper都有什么。

  • int HasTranslucentPolygonalGeometry () override
    在渲染时被调用，返回是否有透明的几何图形。有趣的是这个函数调用了上面的InitPathTraversal（）函数。往下看了看InitPathTraversal（）函数之后，发现这个函数的主要作用并不像官方API参考提到的那样。这个函数初始化了对vtkCollection的当前指向元素指向到vtkCollection的最顶部元素。而这个vtkCollection函数是用来保存一系列的vtk对象的。其中一个用处就是存储vtkPropAssembly的结构路径，即这个vtkProp的组合是以什么结构组成的。这个函数之所以会用到vtkPropAssembly，因为判断是否有透明的几何图形不仅要在一个vtkProp中判断，如果是一个vtkPropAssembly，就要通过对应的vtkCollection来遍历所有的vtkProp来依次判断。因此InitPathTraversal（）函数就是用来初始化vtkCollection的指向位置的。

  • int RenderTranslucentPolygonalGeometry (vtkViewport *) override
    渲染vtkProp中透明的几何部分

  • int RenderVolumetricGeometry (vtkViewport *) override
    渲染vtkProp中的体数据

2.2.4 mitkVtkPropRenderer

这个类的前身是OpenGLRender，现在替换成了这个类。这个类会做一些底层的设置，比较复杂，但是可以看到比较多相对底层的渲染的实现。这个类需要更加仔细得看一看，下面分析一下这个类。 首先是这个类的公有成员：

  • 这个枚举定义了渲染的vtkProp的四种类型，就是之前提到过的不透明、透明、遮挡、体数据。这个枚举会作为参数传递给下面的函数。

  • 这个枚举定义了上面所说的vtkPick的三种类型。可以在.cpp函数里找到对这部分的注释，主要介绍了三种不同类型的区别，第一种速度最快但不支持透明物体，最后一种速度最慢但是效果最好。具体的PickMode的设置是通过宏定义的，不用再往下深究了。

  • typedef std::map MappersMapType; 这个自定义类型是存储Mapper的map，会在下面的函数中用到。

公有函数：

  • 构造函数 mitk::VtkPropRenderer::VtkPropRenderer（…） 参数太多在这里省略，具体可以看.cpp文件。这个构造函数初始化了vtkPicker、vtkLight和vtkRender，同时初始化了一个3D的mapper：PlaneGeometryDataVtkMapper3D，这个mapper可以用来渲染平面的3D图形。只要存在一个VtkPropRender就至少有这个mapper。

  • void SetDataStorage (mitk::DataStorage *storage) override
    设置了用来渲染的Prop存放的DataStorage。

  • int mitk::VtkPropRenderer::Render(mitk::VtkPropRenderer::RenderType type) 使用这个函数来开始MITK的渲染进程。其中的参数就是之前的那个枚举类中的值。值得一提的是在函数的具体实现中其实只实现了Oqaque和Overlayer两个类型的几何图形的渲染。并且Overlayer主要是用来渲染文本的，渲染出一个只包含该文本的图片。

  • void mitk::VtkPropRenderer::PrepareMapperQueue()
    这个函数在上一个函数中被调用了。主要的作用是迭代data tree来找到应该用来渲染的对应的mapper。例如如果data tree的这个节点是个图片，那么就把MappersMapType对应的节点的位置设置为图片的mapper。通过这种方式来对用来渲染这个data tree的所有节点的mapper进行分类。 这个体有一行代码很有意思： const mitk::Mapper::Pointer mapper = node-&gt;GetMapper(m_MapperID); 这个GetMapper函数真正调用的是mitk::Mapper::Pointer mitk::CoreObjectFactory::CreateMapper(mitk::DataNode *node, MapperSlotId id) 这个函数，这个函数判断DataNode的类型从而指定特定的mapper，这个地方是我认为真正为data tree中的每个node分配不同的mapper的位置。 这段代码就是上个函数的一部分，可以看到是如何通过MapSlotid和node的类型来确定使用哪个mapper的。也就是说MITK在渲染的过程中自动分析了要渲染的物体的类型从而指定特定的mapper。3D部分也是差不多的代码。

  • void mitk::VtkPropRenderer::InitRenderer(vtkRenderWindow *renderWindow)
    这个函数是用来初始化renderer的，被构造函数所调用，设置了交互和回调函数。

  • void mitk::VtkPropRenderer::RenderingCallback(vtkObject *caller, unsigned long, void *, void *)
    这个函数被上面的函数所调用，设置了交互的回调函数，具体的交互方式可以通过这两个函数设置。

  • void mitk::VtkPropRenderer::Resize(int w, int h)
    这个函数是用来在窗口大小改变后自动调用的函数，两个参数是新的窗口的宽高。

  • void mitk::VtkPropRenderer::InitSize(int w, int h)
    这个函数是用来初始化窗口大小。

  • int mitk::VtkPropRenderer::WriteSimpleText(std::string text, double posX, double posY, double color1, double color2, double color3, float opacity)
    这个函数是用来渲染简单的文本，参数名字就告诉了我们是什么作用。把文本作为vtkTextActor，并且加入了TextCollection这个map，函数返回了该文本在TextCollection中的位置索引。

  • void mitk::VtkPropRenderer::MakeCurrent()
    调用这个函数尝试让调用的窗口成为调用线程的当前图形环境。

  • void mitk::VtkPropRenderer::PickWorldPoint(const mitk::Point2D &displayPoint, mitk::Point3D &worldPoint) const
    这个函数是用来执行Pick点的操作的，第一个参数是用户在屏幕上点的坐标，第二个参数存储我们Pick到的点。这个函数里有三个判断，分别对应了之前的三种不同的Pick方式，具体方式见上面的介绍。

  • mitk::DataNode *mitk::VtkPropRenderer::PickObject(const Point2D &displayPosition, Point3D &worldPosition) const
    这个函数和上一个类似，但是是用来执行Pick对象操作的，返回的是一个存储了Pick到的对象的节点。

  • void mitk::VtkPropRenderer::ReleaseGraphicsResources(vtkWindow * /renWin/)
    这个函数是用来释放当前窗口的图形资源的。它判断了每个DataNode的节点，如果该节点的数据有一个vtk的mapper，那么说明这是一个用来渲染的图形资源，然后就释放它。

  • void mitk::VtkPropRenderer::PrepareRender()
    在渲染前的一些准备，主要的工作就是判断当前窗口的mapper类型，如果是3Dmapper的话就设置镜头为跟踪球模式（就是我们常用的那种3d交互模式），如果是2D的话就是没什么交互模式。

3.2

如果我们需要新建一个渲染管线，大部分情况下我们可以为这个渲染管线设置独特的交互，和QmitkStdMultiWidget不一样的交互，我们需要实现下面的代码，很好理解。

//新建一个RenderManager
mitk::RenderingManager::Pointer renderingManager = mitk::RenderingManager::New();
// 新建一个DataStorage和Interactior
mitk::DataStorage::Pointer dataStorage = mitk::DataStorage::New();
mitk::GlobalInteraction::Pointer globalInteraction = mitk::GlobalInteraction::New();
//绑定到RenderManager
renderingManager->SetDataStorage( dataStorage );
renderingManager->SetGlobalInteraction( globalInteraction );
// 新建一个QmitkRenderWindow，包含了这个RenderManager
QmitkRenderWindow* renderWindow = new QmitkRenderWindow( parent, "name", renderer, renderingManager )

4. MITK渲染遇到的问题

说到MITK渲染的问题，得回到为什么要分析MITK的渲染。最初的任务是移植Mevislab的一个渲染效果到MITK中，因为MITK集成的VTK和Mevislab都是使用OpenGL的底层，好像可以互相移植一下。在分析了Mevislab中的工程的过程中，发现其实Mevislab的使用是很灵活的，提供了大量的效果的过滤器，有很多基础的显示算法都集成进去了，但是我在VTK中并没有找到相应过滤器，这给我很大的困扰。后来调查了一下Mevislab发现其实它并不是直接使用的OpenGL作为底层，而是使用封装了OpenGL的OpenInventor作为底层，有兴趣的同学可以查一下OpenInventor（OIV）。OIV和VTK的能力显然是不对等的，方向也不是一致的，一个是专业用来渲染的API，一个是数据可视化的API。这就使得很多OIV中的效果无法在VTK中复现。也就很难再MITK中复现。 以上的意思就是如果不扩展VTK，那么MITK的渲染效果其实是有很大的制约，光靠API的调用是无法完成的。我目前在学习OpenGL，其实有些VTK中没有集成的效果我也知道如何用OpenGL实现，但是如何集成到VTK中还是一个障碍，这需要更改VTK的源码，是个很复杂的工作。这个工作我认为其实意义不大，不如直接不用VTK换一个底层API来的方便。 MITK渲染最大的问题就在于VTK的制约，如果无法对VTK有很大的显示效果的提升，那么MITK的显示效果不可能提升。 另外的一个大问题在于MITK的渲染框架太过复杂，上面的类图分析已经展示了它调用的复杂性，对于很多刚接触的人来说简直头大。其实这不利于用MITK进行开发。
